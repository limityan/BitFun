§04.1 / Case · Cursor 12 / 19

案例 03 · Cursor 误删 Railway 生产数据卷2026.04.25 · Production Incident · 全部位于 Harness 缺失

● critical
数据丢失：生产数据卷
根因：约束层缺失
Prompt 中的"建议"无法替代硬拦截
step 01
Agent 自主获取访问凭证
从环境变量与历史会话中提取 Railway API token，缺少最小权限隔离。
step 02
误判执行环境
将 staging 与 production 视为同一调用域，未校验 volumeId 是否跨环境共享。
step 03
构造删除调用
直接拼装 GraphQL volumeDelete(id: ...)，无人工确认环节、无危险命令清单匹配。
step 04
生产数据卷被删除
操作返回成功，数据丢失。事后 Agent 才报告"已删除"。
Agent 事后自白

"I assumed the staging environment API call would apply only to staging. I did not verify. I did not check whether the volume ID was shared across environments."

"我假设 staging 环境的 API 调用只会作用于 staging。我没有验证。我没有检查 volume ID 是否跨环境共享。"

提示词中的"小心"在 token 概率分布面前没有强制力。
约束层必须在执行链路上做硬拦截，而不是写在 prompt 里。
这就是 §05 第 4 步"危险命令硬拦截"必须存在的原因。
Source: Cursor incident postmortem · 2026.04.25
