Harness 方法论如何落到一个 Coding Agent 里。

下面不是 BitFun 架构复盘
而是一个 AI Coding Agent
如何按 Harness 六维设计
Harness landing pattern
BitFun Agent 的设计重点，不是"让模型更会写代码"，而是给每一种编码行为配一层 Harness。

同一个模型，在不同 Harness 下会表现成不同的 Agent：能规划、能调试、能修复、能审查。BitFun 的设计细节可以理解为把 Context / Tools / Guardrails / Feedback / Trace 这些方法论组件逐一产品化。

Context 先让 Agent 看见正确上下文：仓库规则、模块边界、历史会话、当前任务模式。上下文不是聊天记录堆叠，而是可选择、可更新、可继承的工作记忆。
Tools 再把能力暴露成受控工具：读文件、改代码、跑命令、取日志、做 review。Agent 不直接"拥有系统"，而是通过工具边界操作系统。
Loop 最后让结果回到闭环：Plan 要审批，Debug 要 runtime 证据，Review 要独立仲裁，交付要测试与 trace。关键判断必须落到外部信号，不能只落在模型自信里。
Six Harness dimensions in BitFun
01
执行Execution
工具通过 tool registry / transport adapter 统一收口，让所有 Agent 在受控工具箱中工作。
core/agentic/tools · transport/adapters
02
记忆Memory
仓库规则、模块边界、会话历史进入上下文，并自动更新，避免 Agent 每轮重新学习架构。
AGENTS.md · .bitfun/sessions
03
反馈Feedback
lint / test / cargo check / Deep Review 都是外部反馈；模型不能靠一句"looks good"宣布完成。
verification matrix · deep_review_policy
04
编排Orchestration
Sub-Agent: Plan / Default / Debug / Review 不只靠提示词控制行为，而是通过预设状态机+原子能力设定规划行为。
agent prompts · session runtime
05
约束Constraint
写文件、运行命令、调试插桩、危险操作等权限都必须被显式约束；能力开放与风险控制一起设计。
mode prompts · tool permissions
06
评估Evaluation
每轮模型输入、工具调用、结果与 diff 可追踪，方便定位 Harness 失效点，保证原子粒度误操作可回滚；任务结束后通过反思总结自主进化。
.bitfun/sessions/{session_id}
