Debug Mode 把失败信号产品化
BitFun · TDD-inspired debug contract prompts/debug_mode.md · L8–L26
You are now in DEBUG MODE. You must debug with runtime evidence.

Traditional AI agents jump to fixes claiming 100% confidence, but fail due to lacking runtime information. They guess based on code alone. You cannot and must NOT fix bugs this way.

Critical constraints:
· NEVER fix without runtime evidence first
· ALWAYS rely on runtime information + code
· Fixes often fail; iteration is expected.

和 TDD 的关系：TDD 用 Red / Green / Refactor 让失败测试驱动实现。BitFun Debug mode 把这个节奏迁移到排障：先得到可观察的 Red，再做最小修复到 Green，最后清理插桩并保留验证证据。

BitFun Debug Mode · TDD loop for bugs
1
Red: make the failure observable
先列 3-5 个可证伪假设，再设计日志或复现步骤。BitFun 要的不是"我猜到了"，而是能稳定看到失败。
2
Red signal: instrument safely
插入结构化、可回收、不可泄露 secrets 的日志；用 // #region agent log 标记，形成临时测试探针。
3
Run: reproduce through product UI
复现不是一句"你再试试"；UI 承接步骤、按钮和状态，让用户行为变成 Agent 可消费的测试运行。
4
Green: patch only confirmed cause
每个假设先判定 CONFIRMED / REJECTED / INCONCLUSIVE；只有 confirmed cause 才能进入修复。
5
Refactor: clean probes after green
修复后再次复现，对比 before / after。只有 Green 后才能移除临时日志，必要时把用例沉淀进测试或 Debug 规则。
Debug Mode = TDD 的排障化落地：先让失败可观察，再让修复对着失败闭环。
