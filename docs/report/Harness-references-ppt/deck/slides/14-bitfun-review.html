BitFun Reviewer：把"模型自审"改成独立反馈层Feedback · findings must be produced, judged, and fixed separately

对 Agent 来说，review 不是再问一遍
"你觉得自己写得好吗"
而是建立外部质量信号
Why review needs harness
执行者天然看不见自己的盲区。

同一个 Agent 刚刚写完代码，再让它"自审"，很容易只复述自己的实现理由。BitFun 的设计是把 Review 从执行链路里拆出来：发现问题的人、判断问题的人、修问题的人，不是同一个角色。

BAD让实现 Agent 自己宣布 clean，缺少独立反馈源。
BAD单 reviewer 容易只看一个领域，漏掉跨层风险。
BADfinding 没有仲裁，false positive 和真正 bug 混在一起。
BitFun Deep Review loop
Business
需求偏差、边界条件、业务不变量
Architecture
分层依赖、平台边界、长期维护成本
Security
权限、注入、敏感信息与越权路径
Performance
复杂度、N+1、资源消耗、响应路径
Frontend
用户路径、状态管理、可访问性、视觉回归
Verification
测试缺口、验证命令、失败输出回注
Judge
只做仲裁：每个 finding 标记为 validated / rejected / downgraded，合并重复项。
Fixer
只修 validated finding，按 severity 排序，修完重新跑对应验证；不能绕过 Judge。
What becomes feedback
1
Finding 必须可定位
不是"这里可能有问题"，而是指向具体文件、逻辑、风险和复现条件。
2
Judge 才能进入修复
reviewer 提供候选信号，Judge 决定哪些信号进入执行队列。
3
Fix 必须带验证
修复不是改完代码，而是让失败检查、测试或 review finding 关闭。
4
结果进入 trace
finding 来源、仲裁理由、修复 diff、验证结果都保留，成为下一轮 Agent 的可学习反馈。
Review Harness = 把质量信号从执行者手里拿出来：独立发现、独立仲裁、按证据修复。
